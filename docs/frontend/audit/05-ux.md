### ID: UX-001

**Categoría:** UX  
**Severidad:** Alta  
**Archivo:** `src/components/Hero.tsx`, `src/components/About.tsx`, `src/components/ContactInfo.tsx`  
**Línea:** 19 | 7 | 8  

**Problema:** Los componentes públicos retornan `null` cuando los datos están cargando (no hay skeleton/loader). Hero, About y ContactInfo desaparecen completamente mientras se cargan los datos del perfil.

**Impacto:** Mala experiencia en conexiones lentas. La página se ve vacía o parcialmente cargada sin indicación visual de que se están obteniendo datos.

**Recomendación:** Implementar estados de carga con Skeleton components antes de que los datos estén disponibles (el componente `Skeleton` ya existe en `src/components/ui/Skeleton.tsx` pero no se usa en la parte pública).

**Justificación:** Los skeletons ya existen como UI primitives pero no se están utilizando en las secciones públicas.

### ID: UX-002

**Categoría:** UX  
**Severidad:** Media  
**Archivo:** `src/components/ContactForm.tsx`  
**Línea:** 17-29  

**Problema:** Después de enviar exitosamente el formulario de contacto, los campos del formulario no se resetean. Al hacer clic en "back" o navegar, los datos anteriores permanecen.

**Impacto:** El usuario debe recargar la página para enviar otro mensaje. Los datos sensibles (email, mensaje) permanecen visibles.

**Recomendación:** Resetear todos los campos del formulario al estado inicial después de un envío exitoso.

**Justificación:** Es una práctica UX estándar para formularios de una sola vez.

### ID: UX-003

**Categoría:** UX  
**Severidad:** Media  
**Archivo:** `src/pages/admin/Profile.tsx`  
**Línea:** 142  

**Problema:** La página de perfil retorna `null` si `profile` no ha cargado, mostrando una pantalla en blanco sin indicación de carga.

```tsx
if (!profile) return null
```

**Impacto:** El usuario ve una página vacía sin saber si está cargando o falló.

**Recomendación:** Usar el componente `Skeleton` existente para mostrar un estado de carga, no `null`.

**Justificación:** El componente `Skeleton` ya existe en la base de código pero no se utiliza aquí.

### ID: UX-004

**Categoría:** UX  
**Severidad:** Baja  
**Archivo:** `src/components/Footer.tsx`  
**Línea:** 48  

**Problema:** El año del copyright está hardcodeado como `2024`.

```tsx
<p>&copy; 2024 {profile?.name ?? "Portafolio"}.</p>
```

**Impacto:** Mostrará un año desactualizado a partir de 2025.

**Recomendación:** Usar `new Date().getFullYear()` para el año dinámico.

**Justificación:** Es una práctica estándar evitar años hardcodeados en copyrights.

### ID: UX-005

**Categoría:** UX  
**Severidad:** Baja  
**Archivo:** `src/pages/admin/Skills.tsx`, `Education.tsx`, `Technologies.tsx`, `Projects.tsx`, `Services.tsx`  

**Problema:** Las operaciones de eliminación usan `window.confirm()` nativo del navegador que no sigue el diseño de la app y no es personalizable.

**Impacto:** Inconsistencia visual. El diálogo nativo se ve diferente en cada navegador/SO.

**Recomendación:** Implementar un modal de confirmación personalizado que siga el design system.

**Justificación:** `window.confirm()` bloquea el hilo principal y no es personalizable ni accesible.
